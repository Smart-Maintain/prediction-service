# Aerospace Predictive Maintenance AI Module

This repository contains the machine learning models, data processing pipelines, and a production-ready API for predicting the Remaining Useful Life (RUL) of aerospace turbofan engines based on the NASA C-MAPSS dataset.

## Architecture Overview
The model is a dual-head Multi-Task Bi-LSTM neural network. For production, it has been exported to the **ONNX** format for high-speed, lightweight inference. The server is built with **FastAPI** coupled with **ONNX Runtime** (meaning there is zero heavy PyTorch dependency required for the backend).

## 🛠 For Backend & DevOps Engineers

The API is fully containerized and ready to be deployed. The relevant files (`main.py`, `multitask_model.onnx`, `Dockerfile`, `requirements.txt`) are located in the `ai study/work/` directory.

### Deploying via Docker (Recommended)
Deploying via Docker is the easiest way to push this to AWS (ECS/EKS), Render, Railway, or Azure.
```bash
# 1. Navigate to the API working directory
cd "ai study/work"

# 2. Build the Docker image
docker build -t predictive-maintenance-api .

# 3. Run the container
docker run -p 8000:8000 predictive-maintenance-api
```
Once running, the API is available at `http://localhost:8000`. You can test endpoints via the autogenerated Swagger UI at `http://localhost:8000/docs`.

### Running Locally (Without Docker)
```bash
cd "ai study/work"
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

## 🖥 For Frontend Developers & External API Integration

Your frontend applications or external microservices should interact exclusively with the `/predict` endpoint to get real-time engine health telemetry.

### Endpoint
`POST /predict`

**Headers:**
`Content-Type: application/json`

### Request Payload Requirements
The model requires a sliding window representing the previous **30 continuous flight cycles** of an engine.
* **Window Size:** The `window` array must contain exactly 30 sub-arrays.
* **Feature Size:** Each cycle (sub-array) must contain exactly **18 normalized feature values** (operational settings and sensor measurements).
* **Format:** `[30, 18]` 2D array.

**Example Request:**
```json
{
  "window": [
    [0.43, 0.81, 0.22, ...],  // Cycle T-29 (exactly 18 float values)
    [0.44, 0.80, 0.23, ...],  // Cycle T-28
    ...
    [0.55, 0.90, 0.31, ...]   // Current Cycle T-0
  ]
}
```

### Example Response
The API will perform inference via the ONNX engine and return the following JSON object:
```json
{
  "rul_cycles": 45.2,
  "alert": false,
  "confidence": 0.8712
}
```

### Response Object Dictionary:
* `rul_cycles` *(float)*: The predicted Remaining Useful Life of the engine in cycles (capped at 125 for healthy engines).
* `alert` *(bool)*: `true` if the engine is in a critical condition and predicted to fail within the next 30 cycles. Otherwise `false`.
* `confidence` *(float)*: The model's percentage confidence level in the binary `alert` assessment (ranges from 0.5000 to 1.0000).

> **Integration Note:** The raw sensor readings coming from the physical engines (or database) **must be normalized** (from 0.0 to 1.0) before being sent to this endpoint, matching the state of the training preprocessing pipelines.
